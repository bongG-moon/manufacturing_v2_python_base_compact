# Harness Principles

이 문서는 OpenAI의 Harness Engineering 공식 글과 내부 참고 문서에서 공통으로 강조한 원칙을 제조 에이전트 하네스에 맞게 정리한 것입니다.

## 1. 모델보다 구조를 믿는다
- 특정 모델 성능에만 의존하지 않는다.
- 모델이 바뀌어도 유지되는 문서 구조, 평가 기준, 실행 계획을 먼저 만든다.

## 2. AGENTS.md는 짧고, docs는 구조화한다
- `AGENTS.md`는 작업 맵 역할만 한다.
- 실제 지식은 `docs/`로 구조화한다.
- 실행 계획과 참조 문서는 별도 폴더로 분리한다.

## 3. 일관성(Idempotency)을 우선한다
- 같은 질문, 같은 데이터, 같은 규칙이면 비슷한 구조의 결과가 나와야 한다.
- 이를 위해 데이터셋 정의, 도메인 해석, 지표 계산 규칙을 문서로 고정한다.

## 4. 문제를 작은 단위로 나눈다
- 질문 해석
- 데이터셋 선택
- 원본 조회
- 후속 분석
- 답변 생성

위 단계를 분리해서 보면, 어디서 문제가 나는지 추적하기 쉬워진다.

## 5. 문서 없는 규칙은 금지한다
- 달성율, 불량률, 수율 같은 핵심 KPI는 문서화한다.
- 데이터셋과 컬럼 의미도 문서화한다.
- 실제 코드가 문서와 다르면 하네스부터 갱신한다.

## 6. 평가를 기본 기능으로 본다
- 좋은 답변도 중요하지만, 좋은 평가 세트가 더 중요하다.
- 단일 조회, 후속 분석, 다중 데이터셋, 없는 컬럼 요청까지 평가 질문 세트에 포함한다.

## 7. 동기화 흐름도 구조의 일부다
- 하네스는 기준 저장소
- 실제 구현 리포지터리는 소비자이자 피드백 제공자
- 따라서 pull / push 흐름도 하네스 안에 스크립트와 문서로 남겨야 한다
